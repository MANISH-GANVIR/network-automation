GIT REPOSITORY -
--------------
1️⃣ WHAT IS GIT REPOSITORY?
   → Folder that Git uses to track code changes
   → Stores snapshots & metadata
   → Must exist to use Git commands

2️⃣ HOW TO CREATE?
   Command: git init
   → Run once per project
   → Creates hidden .git folder
   → Project now under Git control

3️⃣ BEFORE vs AFTER GIT INIT:

   BEFORE:
   Regular folder → Git commands don't work

   AFTER:
   git init runs → .git folder created → Git commands work ✓

4️⃣ KEY POINT:
   No Repository = Can't use Git
   Repository Created = Can use all Git features

5️⃣ AUTO-INITIALIZED PROJECTS:
   Some tools (React CLI, etc.) auto-create repo
   → Don't need git init again

GIT COMMITS -
-------------
1️⃣ WHAT IS A COMMIT?
   → Code snapshot at a specific point in time
   → Created after meaningful changes
   → Has unique ID & message

2️⃣ TWO-STEP PROCESS:

   STEP 1: git add <filename>
   → Stage files you want in snapshot
   → Choose which changes to include

   STEP 2: git commit -m "message"
   → Create snapshot from staged changes
   → Requires descriptive message

3️⃣ WHY TWO STEPS?
   → Control which files go in which commit
   → Don't have to commit everything at once

4️⃣ COMMON COMMANDS:

   git add index.html          → Stage file
   git status                  → Check staged files
   git commit -m "msg"         → Create commit
   git log                     → View all commits

5️⃣ WORKFLOW:

   Code Changes → git add → git commit → Snapshot Created ✓

6️⃣ COMMIT MESSAGE:
   → Short & descriptive
   → Example: "initial commit", "fix bug"

7️⃣ GIT LOG:
   → Shows all commits in order
   → Displays commit ID & message
   → Lets you go back to any commit

KEY POINT: Commits = Project history | Can rollback anytime

GIT HEAD & CHECKOUT -
---------------------

1️⃣ WHAT IS HEAD?
   → Pointer to currently loaded commit
   → Always points to latest by default

2️⃣ GIT CHECKOUT:

   git checkout <commit-id>
   → Load old commit temporarily
   → Safe - nothing deleted

   git checkout <branch-name>
   → Return to branch with all commits

3️⃣ WORKFLOW:
   Make Changes → git add → git commit
                                 ↓
                            HEAD moves to
                            new commit

4️⃣ WHY CHECKOUT OLD COMMITS?
   • Debug/Find bugs
   • Revert mistakes
   • Review code changes
   • Restore deleted files

5️⃣ COMPLETE TIMELINE:

   Commit-1 → Commit-2 → Commit-3
   (v1)       (v2)       (v3 - Latest)
                          HEAD ↑

   AFTER: git checkout Commit-1-ID

   Commit-1 ← Commit-2 → Commit-3
   HEAD ↑     (Hidden)   (Still safe)

   AFTER: git checkout main

   Commit-1 → Commit-2 → Commit-3
                         HEAD ↑

6️⃣ USE CASE DIAGRAM:

   Bug Found in v3
        │
        ▼
   Checkout v1 (Working) ✓
        │
        ▼
   Compare & Find Issue
        │
        ▼
   git checkout main
        │
        ▼
   Create Fix (New Commit)

KEY POINT: Checkout = Time travel for debugging | Safe always

GIT REVERT:-Create new undo commit | History preserved
----------

1️⃣ WHAT IS GIT REVERT?
   → Undo changes from specific commit
   → Creates NEW commit (doesn't delete old one)
   → Safe way to undo mistakes

2️⃣ COMMAND:

   git revert <commit-id>
   → Reverts that specific commit
   → Prompts for commit message
   → Creates undo commit automatically

3️⃣ KEY DIFFERENCE:

   MANUAL UNDO:
   Remove paragraph → Edit files → git add → git commit
   (More work)

   GIT REVERT:
   git revert <id> → Done ✓
   (Automatic)

4️⃣ HOW IT WORKS:

   Commit-1 (initial) → Commit-2 (added link) → Commit-3 (reverted)
   (paragraph gone)

   Old commits still exist, new undo commit created

5️⃣ TIMELINE DIAGRAM:

   BEFORE REVERT:

   Commit-1 → Commit-2 (paragraph added) → HEAD

   AFTER: git revert Commit-2-ID

   Commit-1 → Commit-2 → Commit-3 (paragraph removed)
                         HEAD ↑

6️⃣ BENEFITS:
   ✓ Automatic undo (no manual editing)
   ✓ Complex changes handled easily
   ✓ History preserved (can see what happened)
   ✓ Safe rollback

7️⃣ WORKFLOW:

   Made Mistake ✗
        │
        ▼
   git revert <commit-id>
        │
        ▼
   Enter message
        │
        ▼
   Undo commit created ✓
   (Original still visible in history)

KEY POINT: Revert = Create new undo commit | History preserved

GIT RESET --HARD - INTERVIEW NOTES

1️⃣ WHAT IS GIT RESET --HARD?
   → Delete commits permanently
   → Go back to specific commit
   → All commits after that are GONE
   → Rewrites history

2️⃣ COMMAND:

   git reset --hard <commit-id>
   → Deletes all commits after specified commit
   → Cannot undo (data loss possible)
   → USE WITH CAUTION

3️⃣ REVERT vs RESET:

   GIT REVERT:
   • Creates new undo commit
   • Keeps history
   • Safe ✓

   GIT RESET --HARD:
   • Deletes commits permanently
   • Rewrites history
   • Risky ✗

4️⃣ TIMELINE DIAGRAM:

   BEFORE RESET:

   Commit-1 → Commit-2 → Commit-3 → Commit-4 → HEAD

   AFTER: git reset --hard Commit-1-ID

   Commit-1 → (Deleted) → (Deleted) → (Deleted)
   HEAD ↑

5️⃣ COMPLETE GIT COMMANDS SUMMARY:

   git init
   → Initialize repository

   git add <file>
   → Stage changes

   git commit -m "msg"
   → Create commit

   git status
   → Check staged files

   git log
   → View all commits

   git checkout <id>
   → Load old commit temporarily

   git revert <id>
   → Create undo commit (safe)

   git reset --hard <id>
   → Delete commits (dangerous)

6️⃣ WHEN TO USE:

   ✓ REVERT = Default choice (safe, keeps history)
   ✗ RESET --HARD = Only if absolutely sure

7️⃣ RISK DIAGRAM:

   Safe Route:
   git revert → New commit → History kept ✓

   Dangerous Route:
   git reset --hard → Data loss → Can't undo ✗

KEY POINT: Use REVERT by default | RESET only with caution

1️⃣ WHAT IS .GITIGNORE?
   → Special file to exclude files/folders from commits
   → Keeps repository clean
   → Prevents auto-generated/personal files being shared

2️⃣ WHY DO YOU NEED IT?

   Example: Your project creates these files:

   __pycache__/
   └─ Function: Python auto-creates cache for faster imports
   └─ Use Case: Speed up code execution on same machine
   └─ Ignore? YES ✗ (recreated automatically on other machines)

   .idea/
   └─ Function: IDE stores editor settings, layout, plugins
   └─ Use Case: Personal editor configuration
   └─ Ignore? YES ✗ (don't share personal preferences)

   *.pyc
   └─ Function: Compiled Python bytecode files
   └─ Use Case: Machine-specific compiled version of .py
   └─ Ignore? YES ✗ (original .py file is sufficient)

   These are AUTO-GENERATED:
   ✗ Not needed in repository
   ✗ Clutters shared codebase
   ✗ Wastes storage space

3️⃣ HOW TO CREATE .GITIGNORE?

   Step 1: In project root, create file named:
           .gitignore

   Step 2: Add files to ignore:

           # Python cache directory
           # Auto-created by Python when imports run
           # Stores compiled bytecode for faster loading
           __pycache__/

           # IntelliJ IDEA IDE settings folder
           # Contains editor configuration, layout, plugins
           # Personal to each developer's machine
           .idea/

           # Python compiled bytecode files
           # Auto-generated from .py source files
           # Machine-specific, recreated automatically
           *.pyc

   Step 3: Save file

GIT BRANCHES - COMPLETE NOTES (ORGANIZED)
-----------------------------------------
Question: What are Git branches?

Answer:
-"A branch is a pointer to a sequence of commits.
Git branches allow parallel development by creating
separate lines of commits that can later be merged
into the main branch.

-By default, a 'main' branch exists automatically
when you initialize a repository. The main branch
is typically developed by the core team and contains
stable, production-ready code.

-You can create new branches (like feature-1) for
developers to work on new features independently
without affecting the main codebase. Each branch
maintains its own commit history, though branches
can share common commits at the branching point.

-Multiple commits can exist in both branches, but
each branch can have unique commits developed by
its assigned developers. This isolation keeps the
main branch stable while feature development happens.

-When development on a branch is complete, you can
merge it back to main using 'git merge', which
combines all changes into a single timeline while
preserving the complete history.
-This enables team collaboration where different
developer teams work on different features
simultaneously without conflicts, keeping the main
branch production-ready and allowing independent
development teams to work in isolation."

Commit:
   └─ Snapshot of code at specific point in time
   └─ Represents a version of project
   └─ Example: C1, C2, C3, C4

Branch:
   └─ Pointer to sequence of commits
   └─ Represents line of development
   └─ Example: main, feature-1, bug-fix

Relationship:
   commit = building block
   branch = container for commits

Example:
   main branch has: C1 → C2 → C3 → C4
   (4 commits organized in 1 branch)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

When you create repository:

   $ git init

Git automatically creates:
   └─ "main" branch (default)

All commits go into main by default:
   └─ Unless you create new branch
   └─ No extra work needed
   └─ Happens automatically

Example timeline:

   $ git init
   └─ main branch created (empty)

   $ git commit -m "C1"
   └─ C1 added to main

   $ git commit -m "C2"
   └─ C2 added to main

   $ git commit -m "C3"
   └─ C3 added to main

   $ git commit -m "C4"
   └─ C4 added to main

Result:
   main: C1 → C2 → C3 → C4

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
DIAGRAM FROM IMAGE - DETAILED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Visual diagram:

                main
                 |
            commit1 (C1)
                 |
            commit2 (C2) ----------- git branch feature-1 ----------->
                 |                                            |
            commit3 (C3)                                  commit2
                 |                                            |
            commit4 (C4) <----------- git merge feature-1 ---- commit5
                 |                                            |
                 ↓                                            ↓

UNDERSTANDING EACH PART:

LEFT SIDE (main branch):
   C1 → C2 → C3 → C4

   What it represents:
   └─ Main branch timeline
   └─ 4 commits in order
   └─ Stable production code
   └─ Developed by main branch developers

TOP ARROW (Create branch):
   git branch feature-1

   What it does:
   └─ Creates new branch from C2
   └─ feature-1 starts at C2
   └─ Both branches share C1, C2
   └─ Separates feature development from main

RIGHT SIDE (feature-1 branch):
   C2 → C5

   What it represents:
   └─ Feature branch timeline
   └─ Started from C2
   └─ Added new commit C5
   └─ Developed by feature branch developers
   └─ Isolated from main branch work

BOTTOM ARROW (Merge branch):
   git merge feature-1

   What it does:
   └─ Takes C5 from feature-1
   └─ Adds to main after C4
   └─ Combines histories
   └─ Executed on main branch (where you want changes)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STEP-BY-STEP FLOW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP 1: Initial commits in main

   main: C1 → C2 → C3 → C4

   What happened:
   └─ Project started with C1
   └─ Added features (C2, C3)
   └─ Fixed bugs (C4)
   └─ All on main branch
   └─ Developed by main branch developers
   └─ C3, C4 are from main branch only

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP 2: Create a new branch

   Command:
   $ git branch feature-1

   Who uses this command:
   └─ Developer who wants to create new feature branch
   └─ Developer currently on main branch

   What happens:
   └─ Git looks at current position (C2 in diagram)
   └─ Creates new pointer named "feature-1"
   └─ Points to same commit (C2)

   Result:
   main:       C1 → C2 → C3 → C4
               (main branch developers continue here)

   feature-1:  C1 → C2
               (feature developers will work here)

   Status:
   └─ Two branches exist
   └─ Both at same history (C1, C2)
   └─ Branches will diverge from C2
   └─ main branch = original development path
   └─ feature-1 branch = new isolated path

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP 3: Two branches now exist

   main:
   C1 → C2 → C3 → C4
   (main branch commits - developed by main developers)

   feature-1:
   C1 → C2 (and will grow here)
   (feature branch commits - will be developed by feature developers)

   What this means:
   └─ Both share C1 and C2
   └─ After C2, development separate
   └─ main has C3, C4 (main branch developers work here)
   └─ feature-1 doesn't have C3, C4 yet
   └─ C3, C4 stay on main only
   └─ feature-1 will add its own commits

SEPARATE DEVELOPMENT DIAGRAM:

Time Point: After Step 3

Developer 1 (on main branch):        Developer 2 (on feature-1 branch):
│                                    │
├─ git checkout main                 ├─ git checkout feature-1
├─ (working on main)                 ├─ (working on feature)
│                                    │
└─ C1 → C2 → C3 → C4                 └─ C1 → C2 → C5 (will be created)
   (main branch)                        (feature branch)

KEY DIFFERENCE:

main branch:
   └─ Developers: Anyone on main branch
   └─ Commits: C1, C2, C3, C4
   └─ Work: Bug fixes, maintenance
   └─ Access: Everyone (production code)

feature-1 branch:
   └─ Developers: Feature team only
   └─ Commits: C1, C2, C5, C6... (will add)
   └─ Work: New payment feature
   └─ Access: Feature team (isolated)

SEPARATION EXPLANATION:

Why separate?
   └─ main branch = stable, tested, production ready
   └─ feature-1 = experimental, under development
   └─ Different developers work on different branches
   └─ No one interferes with production code
   └─ feature can be broken, main stays safe

How it stays separate?
   └─ Different branch pointers
   └─ main points to C4
   └─ feature-1 points to C2 (at this step)
   └─ When feature developers commit, only feature-1 advances
   └─ main branch stays at C4 until merge

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP 4: Work independently

   Developer 1 on main:
   └─ Currently on main branch
   └─ $ git checkout main (confirms on main)
   └─ Continues normal work
   └─ Creates C3, C4
   └─ Stable, tested code
   └─ Fixes bugs, adds maintenance

   Developer 2 on feature-1:
   └─ Currently on feature-1 branch
   └─ $ git checkout feature-1 (switched to feature)
   └─ New feature development
   └─ Creates C5 (with new feature)
   └─ Independent from main
   └─ Adds payment processing

   Result:
   main:       C1 → C2 → C3 → C4
               (main branch only)

   feature-1:  C1 → C2 → C5 (new feature)
               (feature branch only)

   Status:
   └─ Different C5s (different commits, different purposes)
   └─ feature-1 isolated from main
   └─ No interference between developers
   └─ main stays stable
   └─ feature-1 can be messy during development
   └─ main developers don't see feature code
   └─ feature developers don't affect production

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STEP 5: Merge branch

   Command:
   $ git merge feature-1

   Who executes this command:
   └─ Someone on main branch (usually lead/manager)
   └─ After feature is complete and tested
   └─ Brings feature-1 changes into main

   Prerequisites:
   └─ Must be on main branch first
   └─ $ git checkout main (switch to main)
   └─ Then $ git merge feature-1 (merge happens)

   What happens:
   └─ Git compares main (C4) with feature-1 (C5)
   └─ Finds common parent (C2)
   └─ Takes changes from C2→C3→C4 (main path)
   └─ Takes changes from C2→C5 (feature path)
   └─ Combines C3, C4 from main
   └─ Combines C5 from feature-1
   └─ Creates unified history
   └─ No conflicts (different areas edited)

   Result:
   main: C1 → C2 → C3 → C4 → C5 (merged)
   (all changes on main now)

   Status:
   └─ All changes on main
   └─ Both feature-1 and main combined
   └─ Ready for production
   └─ Complete codebase
   └─ Users get payment feature

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
IMPORTANT CONCEPTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

HEAD:
   └─ Pointer to current branch
   └─ Shows where you are working
   └─ Moves when you switch branches

Example:
   $ git checkout feature-1
   └─ HEAD moves to feature-1
   └─ You are now on feature-1 branch
   └─ Any commits go to feature-1

   $ git checkout main
   └─ HEAD moves to main
   └─ You are now on main branch
   └─ Any commits go to main

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Shared commits:
   └─ C1, C2 exist in both branches
   └─ Same data, same ID
   └─ Not duplicated, just referenced
   └─ Branching point starts here

Example:
   main:       C1 → C2 → C3 → C4
               └─────┬─────┘
   feature-1:      C2 (same C2)
                   └→ C5

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Independent commits:
   └─ C5 on feature-1 ≠ C3, C4 on main
   └─ Different code
   └─ Separate development paths
   └─ Different purposes

Example:
   feature-1 C5: Add payment processing
   main C3: Add database
   main C4: Fix login bug

   (All different, no conflict)
   (C3, C4 on main only - feature doesn't have them)
   (C5 on feature only - main doesn't have it yet)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Branch behavior:
   └─ Does NOT affect other branches until merge
   └─ Isolation: work safely
   └─ Protection: main stays stable

Example:
   feature-1 breaks? → main still works
   feature-1 deleted? → main unaffected
   feature-1 merged? → main gets changes

COMMANDS REFERENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create branch:
   $ git branch feature-1
   └─ Creates new branch from current commit
   └─ Doesn't switch to it
   └─ You stay on current branch
   └─ New branch is for other developers

Switch branch:
   $ git checkout feature-1
   └─ Moves HEAD to feature-1
   └─ Updates working files
   └─ You can now work on feature-1
   └─ Commits will go to feature-1

Merge branch:
   $ git merge feature-1
   └─ Combines feature-1 into current branch
   └─ Must be on main branch first
   └─ Preserves all commits
   └─ Brings feature code to main

List branches:
   $ git branch
   └─ Shows all branches
   └─ * marks current branch
   └─ Helps you see all available branches

Delete branch:
   $ git branch -d feature-1
   └─ Removes branch pointer
   └─ Commits still exist in history
   └─ Usually done after merging

   (Done by: Feature developers)
   (Purpose: Start working on feature)

   (Purpose: Add code to feature branch)

   git merge feature-1

   git branch -d feature-1
   (Done by: Maintainer)
   (Purpose: Clean up)

WHAT ARE GIT BRANCHES?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Branch = Separate line of development
   └─ Multiple developers work simultaneously
   └─ Main code stays stable
   └─ Features developed in isolation

Why use branches?
   └─ Dev 1: Works on main (production code)
   └─ Dev 2: Works on feature (new feature)
   └─ No conflicts
   └─ Easy to merge when done

Real example:
   main branch = Working website
   feature branch = Payment system (under development)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
BASIC CONCEPTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Commit:
   └─ Snapshot of code
   └─ Has ID (C1, C2, C3, C4)
   └─ Can't be changed, only viewed

Branch:
   └─ Pointer to commits
   └─ Has name (main, feature-payment, etc.)
   └─ Can be created, switched, deleted

Main Branch:
   └─ Production code
   └─ Always stable
   └─ Protected (needs approval to merge)

Feature Branch:
   └─ New feature development
   └─ Isolated from main
   └─ Can be broken during development

Merge:
   └─ Combine two branches
   └─ All commits from feature come to main

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

If both edited SAME lines (conflict):

   Main: Changed line 5
   Feature: Also changed line 5

   Result: CONFLICT ✗

   What Git shows:
   <<<<<<< HEAD
   Welcome!  (from main)
   =======
   Welcome   (from feature)
   >>>>>>> feature-restructure

How to resolve:
   └─ Manual: Choose which to keep
   └─ Edit file, remove conflict markers
   └─ Save and commit
   └─ Advanced topic (not covered here)

This example: NO conflicts (different lines) ✓

KEY MERGE CONCEPTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Merge = Combining two branches

What happens:
├─ Git finds common parent (C4)
├─ Git identifies what changed in each branch
├─ Git tries to combine changes
├─ If no conflicts: Auto-merge ✓
└─ If conflicts: Manual resolve needed

Merge commit:
├─ New commit (C5')
├─ Has two parents (both C5s)
├─ Contains both changes
└─ Preserves history ✓

Feature branch after merge:
├─ Remains unchanged
├─ Still has old commits
├─ Can be deleted safely
└─ Changes preserved in main

One-way merge:
├─ We merged feature INTO main
├─ Feature branch untouched
├─ Could also merge main INTO feature
└─ But not needed here

Auto-merge requirement:
├─ Changes in different parts
├─ OR different files
├─ Git can apply both
└─ No manual intervention

Merge message:
├─ Auto-generated by Git
├─ Describes merge
├─ Can be custom
└─ Save with editor
